# 单子分析与项目初始化

你是一个需求分析专家。根据用户提供的单子描述，完成以下工作。

## 输入

单子描述：`$ARGUMENTS`

- 如果 `$ARGUMENTS` 是一个 URL，先使用 WebFetch 获取需求内容。
- 如果 `$ARGUMENTS` 是一个 `.doc` 或 `.docx` 文件路径，先用 LibreOffice 转为 PDF，再转为图片让 Claude 直接看到完整排版：
  ```bash
  # 1. Word → PDF（使用 docx skill 的 soffice.py，没装 docx skill 就改用 soffice/libreoffice 直接调用）
  python "${CLAUDE_SKILL_DIR}/../docx/scripts/office/soffice.py" --headless --convert-to pdf "<文件路径>"
  # 2. PDF → 图片（使用 PyMuPDF）
  PYTHONIOENCODING=utf-8 python "${CLAUDE_SKILL_DIR}/scripts/pdf2images.py" "<pdf路径>"
  ```
  脚本会在同目录下生成 `page_1.jpg`、`page_2.jpg` 等图片，然后用 Read 工具读取图片（Claude 多模态识别）。
- 如果 `$ARGUMENTS` 是一个 `.pdf` 文件路径，同样转为图片后用 Read 读取，确保 Claude 能看到完整排版：
  ```bash
  PYTHONIOENCODING=utf-8 python "${CLAUDE_SKILL_DIR}/scripts/pdf2images.py" "<pdf路径>"
  ```
  如果文档页数较多（>10页），可以先只转前几页快速浏览，再按需转其余页。
- 如果 `$ARGUMENTS` 是图片文件路径（`.png`、`.jpg`、`.jpeg`、`.bmp`、`.webp`），直接用 Read 工具读取图片（Claude 支持多模态识别图片内容）。如果有多张图片，可以一次性并行读取。

## 第一步：需求分析

仔细阅读单子描述，提取以下关键信息：
- **核心需求**：用户到底要做什么
- **技术栈**：涉及哪些语言、框架、工具
- **交付物**：最终要交付什么（代码/文档/部署等）
- **约束条件**：有没有特殊要求（平台、版本、性能等）

如果信息不足，使用 AskUserQuestion 向用户询问关键细节。

## 第二步：历史项目匹配（可选，依赖个人记忆系统）

> 如果你没有搭建 memsearch 或 project-log.md，可以跳过本步骤直接进入难度评估。

从第一步提取的技术栈和核心需求中提取关键词，搜索历史项目经验。这一步的目的是用过去的真实数据来校准后续的难度评估和报价，避免重复踩坑或低估难度。

### 搜索流程

1. **提取搜索关键词**：从核心需求和技术栈中各提取 1-2 个关键词（如 "OpenCV"、"图像处理"、"React dashboard" 等）

2. **调用 memsearch 多轮搜索**（用 Bash 工具执行）：
   ```bash
   # 按技术栈搜索
   python ~/.claude/memsearch/memsearch.py search "<技术栈关键词>"
   # 按需求类型搜索
   python ~/.claude/memsearch/memsearch.py search "<需求类型关键词>"
   ```
   可以并行执行多个搜索命令以加快速度。

3. **读取完整项目记录**：同时读取 `~/.claude/memory/project-log.md`，从中查找与当前需求相似的项目。

### 提取信息

从搜索结果和项目记录中，提取每个相似项目的：
- 项目名称和需求简述
- 客户报价和实际到手金额
- 实际开发耗时
- 最终结果（成功/失败）
- 踩过的坑和经验教训

### 输出格式

如果找到匹配项目，在分析输出中展示：

```
### 历史项目参考

| 项目 | 金额 | 耗时 | 结果 | 相关度 |
|------|------|------|------|--------|
| <项目名> | ¥<金额> | <天数> | <成功/失败> | <高/中/低>（<原因>） |

⚠️ 风险提示：<如有失败项目或踩坑经验，在此列出具体警告>
💡 定价参考：<基于历史数据的报价建议，如"类似需求上次报 ¥180 做了 8 天亏损，建议本次至少 ¥500+">
```

如果 memsearch 和 project-log.md 都没有匹配结果，跳过本步骤，直接进入难度评估。

## 第三步：难度评估

根据以下维度对单子进行评分（每项 1-5 分）：

| 维度 | 说明 |
|------|------|
| 技术复杂度 | 涉及的技术难度，是否需要深入了解某领域 |
| 工作量 | 预估代码量和文件数 |
| 不确定性 | 需求是否清晰，是否有模糊地带 |
| 调试难度 | 问题排查和测试的难度 |
| 集成复杂度 | 是否涉及多系统/多模块交互 |

根据总分给出难度等级：
- **简单** (5-10分)：基础任务，常规实现 → 报价 ¥50-200
- **中等** (11-17分)：需要一定经验，有些技术点需要研究 → 报价 ¥200-800
- **困难** (18-22分)：复杂任务，需要深入研究或大量代码 → 报价 ¥800-2000
- **极难** (23-25分)：高度复杂，涉及多个高难度技术点 → 报价 ¥2000+

**历史数据校准**：如果第二步找到了相似的历史项目，用历史数据来校准上述报价范围。比如某个"中等"难度的需求，历史上类似项目实际花了 8 天且失败了，那就不能按 ¥200-800 报价，应该上调到困难档位甚至更高。历史项目的实际耗时和成败是最真实的参考依据。

### 平台抽成与到手金额（按你自己常用平台改）

> 下面是 hjn 常接的平台和抽成比例，只做参考，请按你自己情况修改。

| 平台 | 服务费 | 到手比例 |
|------|--------|----------|
| 普合 | 30% | 70% |
| 明思 | 30% | 70% |
| 猪八戒 | 40% | 60% |
| 优创 | 40% | 60% |
| 红太狼 | 40% | 60% |
| 知掘 | 40% | 60% |
| 私人单 | 0% | 100% |

报价时需同时展示：
- **客户支付金额**（报价）
- **平台抽成**
- **实际到手金额** = 报价 × 到手比例

如果尚未确定平台，按 40% 抽成（最差情况）估算到手金额。

## 第四步：确认是否接单

分析完成后，使用 AskUserQuestion 询问用户："这单接不接？"

- 如果用户选择**不接**，到此结束，不创建任何文件夹。
- 如果用户选择**接**，继续收集订单信息。

接单后，使用 AskUserQuestion 一次性收集订单信息(最多4个问题)：

### 来源平台
固定选项：`猪八戒` / `普合` / `明思` / `优创` / `红太狼` / `知掘` / `私人单`（按你自己情况改）
- （可选）如果输入来自 wecom-monitor 项目（路径包含 `wecom-monitor/output/`），读取 `ocr_result.txt` 或 `panel_row.png` 获取平台来源，将识别到的平台放在选项第一位作为默认选项。这是 hjn 的企业微信监听项目，没有的话直接忽略。
- 如果无法识别平台，按默认顺序展示选项

### 其他信息
- **订单编号**：平台上的订单号（可选"暂无"）
- **金额**：用户给出的是客户支付的报价金额
- **截止时间**：交付截止日期（可选"待定"）

### 金额计算

用户提供的金额是客户报价（客户支付金额），需要根据已选平台的抽成比例自动计算实际到手金额：
- 普合/明思：到手 = 报价 × 70%
- 猪八戒/优创/红太狼/知掘：到手 = 报价 × 60%
- 私人单：到手 = 报价 × 100%（无抽成）

**Obsidian 订单记录和 README.md 中的金额字段一律写实际到手金额**（取整），同时在备注中注明客户报价原始金额。

收集完成后，将这些信息记录到项目的 README.md 顶部的订单信息区，同时在 Obsidian 接单文件夹创建订单记录，然后继续第五步创建项目。

**商家字段映射**：用户说的来源企业需要映射到你在第三步里列出的标准值。如果不在列表中，使用原始名称。

## 第五步：创建项目文件夹

根据需求内容自行命名项目文件夹（英文短名，简短为主），然后在 `<YOUR_FREELANCE_DIR>` 下创建：

### 目录结构

```
<YOUR_FREELANCE_DIR>/<项目名>/
├── client/               # 客户文件（客户提供的原始文件，文件名不可修改）
├── src/                  # 源代码
├── docs/                 # 自己的需求文档、参考资料
├── README.md             # 项目说明
├── CLAUDE.md             # Claude 开发指南
└── .gitignore            # Git 忽略规则
```

**注意**：`client/` 文件夹用于存放客户提供的所有原始文件（需求文档、数据、图片等），这些文件名必须保持原样，不可重命名。

**自动归档**：如果 `$ARGUMENTS` 是本地文件路径（PDF、DOCX、DOC、图片等），在创建项目文件夹后，自动将原始文件复制到 `client/` 目录下：
```
cp "<原始文件路径>" "<YOUR_FREELANCE_DIR>/<项目名>/client/"
```
如果 `$ARGUMENTS` 包含多个文件路径（空格或逗号分隔），则全部复制。保持原文件名不变。

### 分析报告.md（仅在用户明确要求时生成）

## 第六步：输出确认

最后展示：
- 项目完整路径
- 文件夹结构树（使用 `tree` 或手写）
- 根据难度等级给出 GSD 开发建议（见下方）

### GSD 开发建议

根据第三步评估的难度等级，给出对应的下一步建议：

| 难度 | 建议 |
|------|------|
| **简单**（5-10分） | 💡 小单直接开干：`/gsd:fast "需求简述"` |
| **中等**（11-17分） | 💡 建议完整规划：`cd <YOUR_FREELANCE_DIR>/<项目名>` → `/gsd:new-project` |
| **困难**（18-22分） | 💡 建议完整规划：`cd <YOUR_FREELANCE_DIR>/<项目名>` → `/gsd:new-project` |
| **极难**（23-25分） | 💡 强烈建议完整规划 + 研究：`cd <YOUR_FREELANCE_DIR>/<项目名>` → `/gsd:new-project`（选择启用 domain research） |

说明：中等及以上的单子都建议走完整 GSD 流程（new-project → plan-phase → execute-phase），需求拆分清晰、进度可追踪、交付有验证。`/gsd:quick` 适合项目进行中的临时小任务，不作为接单的主入口。

将建议以可复制的命令格式展示，方便用户直接粘贴执行。不要询问是否用 VSCode 打开项目。
